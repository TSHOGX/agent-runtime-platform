# Phase 9 Code Map

This map records the pre-Phase 9 worktree facts that the Phase 9 slices changed.
It is retained as implementation history; the live baseline is summarized in
[current-status.md](../current-status.md).

Migration scope follows [README.md](./README.md) and
[implementation-slices.md](./implementation-slices.md).

| Area | Pre-Phase 9 code fact | Phase 9 owner | Target |
| --- | --- | --- | --- |
| Agent constants | `orchestrator/internal/agents/agents.go` defines only `claude`, `sh`, `ProtocolClaudeStreamJSON`, and `ProtocolShellPTY`. | 9a, 9b | Add canonical `claude_code`, remove `claude` from new runtime selectors, and promote temporary 9a facts into `DriverSpec`. |
| Driver registry | No `DriverSpec` structure exists. | 9b | Add canonical driver/provider registries; do not register `claude` as an alias. |
| Generic config | `orchestrator/internal/config/config.go` exposes `ClaudeConfig` and `ModelProxyConfig`; generic `harness.agents`, `harness.model_profiles`, and `harness.runtime_providers` are absent. | 9c | Replace legacy `claude:` config with generic agent/model/runtime-provider config. |
| Model proxy alias | `harness.model_proxy` exists and is mirrored into legacy Claude config. | 9c | Keep `harness.model_proxy.sandbox_base_url` as the only sandbox proxy alias source; profiles reference it by `proxy_ref`. |
| Runtime profile schema | `agent_runtime_profiles.agent` has a SQL check for `('claude','sh')`; profile identity includes Anthropic-named model proxy/secret columns. | 9a, 9b, 9f | Rebuild `agent_runtime_profiles` with canonical `driver_id`, model-proxy field names, and a replacement uniqueness index; 9b moves validation to registries; 9f widens the constrained canonical-driver slots for `pi` while preserving valid post-9a rows. |
| Sandbox contract validation | `orchestrator/internal/store/sandbox_contract.go` accepts only `contract_schema_version == 1`; `sandbox_contracts` persists only the outer contract version. | 9a, 9c | Persist payload schema and gate versions, add one v2 loader, delete/fail closed on v1 rows after cutover, reject new v1 writes, and use `contract_gate_version` to distinguish valid pre-9c nullable-digest rows from invalid 9c+ rows. |
| Sandbox contract builder | `orchestrator/internal/server/server.go` builds a v1 contract with string `driver`, top-level `runtime_adapter`, runsc facts, Linux `forbidden_capabilities`, and Phase 8 `data_volumes` evidence. | 9a | Build v2 payloads with driver/provider objects, DataVolume evidence, versioned digests, snapshot policy, and credential policy. Full shape: [contract/target-shape.md](./contract/target-shape.md). |
| Schema-pack mount | `orchestrator/internal/runtime/runtime.go` and `mount_plan.go` auto-mount repo-root `schema-pack` at `/schema-pack` when the directory exists. | 9a | Disable or gate this mount for Phase 9 v2 launches. Phase 9 v2 has no `input_digests.schema_pack_digest`, mount evidence field, or validation preimage for schema-pack content; native driver schema packs remain Phase 10 adapter evidence unless a later Phase 9 gate defines them. |
| Proxy authorization | `orchestrator/internal/store/proxy.go` reads current credential policy and contract fields for model-proxy authorization. | 9a, 9c, 9e | Authorize through the shared v2 contract loader using driver/provider identity and explicit grant allowlists. |
| Control manifest | `orchestrator/internal/runtime/runtime.go` projects `agent`, `claude_session_uuid`, `resume_claude`, and `claude_code_disable_nonessential_traffic`. | 9a, 9d | 9a adds driver/provider projection and generated driver config path, but keeps the existing protocol-v1 fields as projection-only compatibility until 9d. 9d removes them after the runner reads the generated driver projection and bridge protocol v2 is required. |
| Driver state | `sessions.claude_session_uuid` is the only durable driver-private session state. | 9a, 9f | Add generic `session_driver_states`, turn-completion CAS, and checkpoint fencing. Generic rules: `driver-state.md`; Pi payloads: `pi-driver.md`. |
| Driver homes | `session_driver_homes(session_id, driver)` is the authoritative persistent `/agent-home` source; `server.go` provisions it with `session.Agent`, so existing rows may be keyed by `claude`. | 9a, later phase | Runtime paths read `sessions.driver_id` and resolve canonical `driver_home_key` explicitly; `claude_code` uses the `claude_code` key. The 9a destructive cutover deletes old `driver='claude'` rows rather than preserving them. |
| Bridge completion | `ack_turn_completed` carries only status/error, and `CompleteTurnParams` has no driver-state payload. | 9a | Add host-only `driver_state_update` and strip private state before public persistence/replay. |
| Host turn input | `runtime.writeUserTurn` chooses Claude vs shell input frame by protocol. | 9d | Host bridge sends driver-neutral `RunTurn` only after bridge protocol v2 negotiation succeeds against persisted image-manifest evidence; incompatible v1 generations fail closed on reconnect/restore. |
| Sandbox runner | `sandbox-image/files/usr/local/bin/harness-agent-entrypoint` branches on `HARNESS_AGENT=claude`, and `sandbox-image/files/usr/local/bin/harness-bridge-client` sends `hello.payload.protocol_version: 1` and uses `make_turn_runner(agent)` with explicit `claude` and `sh` branches. The Claude runner currently treats `not self.first_turn` and `claude_session_initialized(agent_home, session_uuid)` as resume selectors in addition to `CLAUDE_RESUME`. | 9a, 9d, 9f | 9a chooses the protocol-v1 projection exception: canonical `claude_code` is projected to sandbox-visible `agent: "claude"` / `HARNESS_AGENT=claude` for these current scripts, while host selectors and persisted evidence stay canonical. 9a patches the current Claude runner so the sidecar-derived `resume_claude` / `CLAUDE_RESUME` projection is authoritative and runner-local `first_turn` or filesystem probes cannot override the command-plan variant. 9d replaces the runner with an `AgentRunner` registry and bridge protocol v2 hello fields, then removes the legacy projection fields; 9f adds Pi after strict grants. |
| Output parsing | `orchestrator/internal/server/stream_parser.go` mixes Claude stream-json parsing, shell events, and raw text fallback in one parser. | 9d | Replace with an `OutputNormalizer` registry and prove native event input with a test-only probe. |
| Model access defaults | `orchestrator/internal/store/resources.go` defaults `model_access_allowed` and provider credential posture from `agent == "claude"`. | 9a, 9c, 9e, 9f | Derive access from `sessions.driver_id`, driver spec, model profile, runtime provider, and strict grants. |
| Session creation API | `orchestrator/internal/server/server.go` decodes `POST /api/sessions` as `agent`, validates raw `claude` / `sh`, defaults omitted input to `cfg.DefaultAgent`, currently has no product capability route, and currently seeds a Claude UUID for every session. | 9a, 9c | 9a keeps a request/config-boundary compatibility adapter that translates legacy `agent` input and `HARNESS_DEFAULT_AGENT=claude` defaults to canonical persisted `sessions.driver_id`, while the destructive cutover deletes old `sessions.agent` rows instead of backfilling them. If 9a.1 ships before the cutover, old rows keep a quarantined legacy-row selector until 9a.4 so nullable `driver_id` staging does not strand them; post-cutover runtime paths do not treat `claude` as an alias. 9a stops using `claude_session_uuid` as a selector. 9c makes product `mode` canonical, adds product-safe `GET /api/deployment-capabilities`, validates session creation against the same capability resolver, and removes/rejects `agent` input. |
| Public session DTO | `orchestrator/internal/server/session_dto.go` returns `agent` from the store row and exposes `active_generation_id`; `frontend/lib/types.ts` models only `agent: string`. | 9a, 9c | 9a may emit legacy `agent` labels only as an API DTO compatibility shim and must not expose raw canonical driver IDs. 9c returns `mode` / `mode_label` and removes `agent` mirrors plus runtime generation IDs such as `active_generation_id`. |
| Frontend picker | `frontend/lib/agents.ts` hardcodes `RuntimeAgent = "claude" \| "sh"` and maps them to `Agent`/`Shell`. | 9c | Fetch `GET /api/deployment-capabilities`, create sessions by product `mode`, read product DTO fields, hide/disable Shell from the product-safe capability DTO, and never expose raw driver IDs. |
| Rootfs driver content | The code does not yet gate configured drivers against an image manifest. | 9c, 9d, 9f | 9c adds the generic image-manifest gate and bridge protocol evidence fields; 9d consumes those fields for bridge protocol v2 reconnect/restore gating; 9f adds Pi-specific manifest fields. |
| Phase 10 docs | Phase 10 states it builds on Phase 9, but sub-targets still need adapter-based implementation. | Phase 10 | Consume Phase 9 driver/provider adapters instead of adding Claude-only paths. |
