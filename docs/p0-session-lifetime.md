# P0: Session and Runtime Lifetime Separation

> Status: planned. Blocks Phase 9 work that depends on long-lived sessions.
> Roadmap entry: [PLAN.md → P0 fixes](./PLAN.md#p0-session-and-runtime-lifetime-separation).

## Goal

User sessions, conversation history, and workspace files must persist effectively forever. Live gVisor runtime resources (sandbox processes, netns/veth, checkpoint images, `/30` slots) must be flexibly releasable and reloadable on a separate lifecycle.

The two lifetimes are conceptually independent but the current code still couples them in several places. This document specifies what to unwind so the rest of the roadmap can build on long-lived sessions.

## Background: What Is and Is Not Already Decoupled

Already decoupled or correctly scoped:

- `messages` and `artifacts` SQLite rows are never deleted by sweep/reaper.
- Workspace directory `/var/lib/harness/sessions/<id>` and agent-home directory `/var/lib/harness/agent-homes/<id>` are never touched by sweep/reaper.
- `MonitorIdleSessions` acts only on generations.
- `harness.reaper.failed_retention` is already generation-scoped rather than session-scoped. P0 still needs to close the cleanup coverage gap described below before checkpoint-image retirement can rely on it.
- Event log retention (`harness.events.retention_*`) is independent of conversation resumability; resumability only requires `messages` and workspace.

Currently coupled (must be fixed):

- `SweepExpiredSessions` at `orchestrator/internal/store/resources.go:559` cascades session expiry into generation failure and reclaimable allocations. The session row becomes `destroyed` and `sessionstate.CanAcceptInput` rejects all further input via the API.
- `failGenerationBeforeTurn` at `orchestrator/internal/server/server.go:440` unconditionally calls `FailSession` on any generation start failure, leaving the session permanently terminal even when conversation history and workspace are intact.
- `handleBridgeCompletion` at `orchestrator/internal/server/server.go:956` treats any failed or canceled `ack_turn_completed` as `session.failed`. The bridge client uses that ordinary turn status for Claude stream errors and nonzero exits at `sandbox-image/files/usr/local/bin/harness-bridge-client:397`, so one agent error can make the session permanently unusable.
- `reserved_checkpointed` allocations are explicitly skipped by the reaper, so long-idle checkpointed sessions accumulate checkpoint images and `/30` network slots indefinitely. Releasing those rows is not enough by itself: as long as the active generation remains `checkpointed`, `ensureActiveGeneration` will try the restore path first, and the restore CAS requires `reserved_checkpointed` allocation/resource rows.
- `runtime.DestroyGenerationResources` at `orchestrator/internal/runtime/runtime.go:348` currently deletes nft/veth/netns only. Per-generation filesystem paths, including `checkpoint`, `control`, `runtime` bundle, `bridge`, and `logs`, are still real directories built from `orchestrator/internal/store/resources.go:1791` and would leak even after resource rows become reclaimable.

## Required Changes

### 1. Hard rename `harness.session_ttl` → `harness.session_retention`

`session_ttl` overloads two distinct lifetimes. The new name expresses intent: how long the session row, history, and workspace persist. `0s` is the new default and means no automatic expiry. No backwards-compat alias — lab configs migrate in one step.

`harness.checkpoint.idle_threshold` already drives generation idle lifetime separately and does not change.

Touch points:

If `HARNESS_SESSION_TTL` is present, startup must fail with an explicit "use HARNESS_SESSION_RETENTION" error instead of silently ignoring the old variable. YAML strict decoding catches old `session_ttl`, but env vars do not have a strict decoder; silently ignoring the old env would make old deployments fall back to the new `0s` no-expiry default.

### 2. Decouple generation start failure from session failure and terminal UI events

`failGenerationBeforeTurn` currently calls both `FailGeneration` and `FailSession`. Generation start failures (probe failure, manifest mismatch, OCI bundle errors) are recoverable by retrying with a fresh generation, but today they leave the session permanently terminal.

Required behavior:

- Direct pre-enqueue generation start failure: mark the generation `failed` and leave the session in an input-acceptable non-checkpointed status (`created` if no successful runtime ever existed, otherwise `running_idle`) so the next user message triggers a cold-start allocation via the existing `ensureActiveGeneration` failed-generation path at `server.go:495`. This rule applies only to the synchronous `sendMessage` path before `EnqueueTurnMessage` has created a queued turn for the submitted input.
- Start-failure finalization must not strand a generation just because prepare/probe/start returned after `lease_expires_at`. Today `FailGeneration` is guarded by `lease_expires_at > now` at `orchestrator/internal/store/controlplane.go:887`, so a slow start failure can leave the active generation lifecycle-busy until maintenance. Chosen contract: start paths must renew the owned generation lease while start is in progress, and the retryable start-failure helper is the only non-recovery helper allowed to finalize after lease expiry. The implementation must wrap all long start phases with the same owned renewal guard: `PrepareGeneration`, artifact digest recording, `MarkGenerationResourcesLive`, `Runtime.Start` including probe/restore work, and the restore-success live CAS. The renewal guard must keep the lease alive for both successful and failed starts, stop after the final success/failure transition, and treat owner loss as stale ownership rather than continuing to mutate the generation. The retryable start-failure helper may fail/reclaim the same generation under an expired lease only when the persisted `lease_owner` still matches this process's start-attempt owner, the generation is still the session's active generation, and the lifecycle state is still the same start attempt (`allocating`/`starting`/`probing` or restore/checkpoint lifecycle as applicable). If recovery, another owner, or generation N+1 has already moved the row, return a stale/lost-ownership result without mutating anything. The direct pre-enqueue variant must also move the session to the retryable non-checkpointed state in the same transaction; the background queued-turn variant must leave the session input-blocking as described below.
- Background cold-fallback generation start failure: mark the replacement generation `failed`, but do not move the session to an input-acceptable status while `ListColdFallbackSessions` still reports queued turns for that session. `startColdFallbackSessions` can call `startEnsuredGeneration` for sessions that already have queued turns at `orchestrator/internal/server/server.go:719`; if that replacement start fails, the session must remain non-terminal and input-blocking, normally `running_active`, until maintenance either starts a replacement generation for those queued turns or explicitly cancels them. Publishing `running_idle` or otherwise enabling new input while old queued work remains is invalid.
- Session-level invariant violation (corrupt session row, store-level CAS impossibility): still fail the session. This is the rare path and the only one that should call `FailSession`.
- Event contract: retryable generation start failures must not publish `session.failed`, and must not publish today's terminal `session.error` event shape unless the frontend is changed to treat it as non-terminal. Prefer adding a non-terminal event such as `generation.error` or `session.runtime_error` with `{terminal:false, generation_id, error_class, error}`; keep `session.error` + `session.failed` only for true terminal session failures.
- Durable state/event contract: retryable/runtime events that affect user-visible retry state must be durable event rows, not hub-only broadcasts. The state transition and the durable event append must happen in the same store transaction for `generation.error`/`session.runtime_error`, restore-fallback retirement, checkpoint-retired, and failed/canceled `ack_turn_completed` outcomes. Store helpers must return the committed event IDs and fresh committed state; server code may publish only from those committed event IDs after the transaction commits. If an implementation deliberately uses one durable event instead of a separate status event, that event payload must fully encode the committed status effect (`session_status`, `session_marked_idle`, `session_last_activity_at`, `active_generation_id`, as applicable) so live SSE and replay reduce identically. Hub-only events are allowed only for best-effort diagnostics where reconnect recovery is an API refresh. The frontend reducer must handle the same typed events from live SSE and replay without treating non-terminal errors as session failures.
- Restore fallback is part of the same contract. `failGenerationForRestoreFallback` at `orchestrator/internal/server/server.go:475` currently fails the generation and publishes `session.error` with `{fallback:"cold"}`, but it does not clear `sessions.status = checkpointed` or `sessions.checkpoint_path`. Replace that with a store-side restore-fallback retirement transaction that mirrors checkpoint-retention retirement for the claimed checkpointed generation: mark the old generation `failed`/reclaimable, leave it as the active failed generation until the N+1 allocation CAS wins, set the session to an input-acceptable non-checkpointed status such as `running_idle`, clear all API-visible checkpoint/restore metadata, and append the durable resulting-status event plus the non-terminal runtime event in the same transaction. Publish the returned committed event IDs before attempting N+1. If replacement allocation or start then fails before enqueue, the session must remain non-checkpointed and retryable rather than stuck `checkpointed` with a failed active generation. Preserve the existing live-runtime cleanup ordering for the restore-start-success / `MarkGenerationResourcesLive` failure path at `orchestrator/internal/server/server.go:429`: once restore has started a container, destroy that runtime synchronously before the store-side retirement marks rows `reclaimable` or allocates generation N+1. The destroy call must use the actual runsc restore/container ID created by restore/start, not the bare session ID. In the current runtime this is `session.RestoreID` / `phase3-<sessionID>` because `Runtime.Start` constructs `phase3-<sessionID>` for restore; if the implementation moves to a generation-scoped restore ID, that ID must be threaded through restore/start and cleanup together. `runtime.Destroy(ctx, session.ID)` is invalid for this path because `Destroy` deletes by runsc container/restore ID. If runtime destruction fails, abort retirement and leave DB rows non-reclaimable rather than recording failed/reclaimable state while a restored container may still be running.
- Expired-runtime recovery is part of the same contract in both startup and periodic maintenance. `RecoverAllocations` currently fails expired `restoring`/`checkpointing` generations and also expired `active`/`idle` generations on the `unknown_after_ack_started` and reconnect-grace paths, then makes their resource rows reclaimable without repairing the owning session or proving that the corresponding runsc container is gone. Replace the mutating `RecoverAllocations` call everywhere it is used, including `orchestrator/cmd/orchestrator/main.go` startup recovery and the periodic `RunPhase7Maintenance` call at `orchestrator/internal/server/server.go:791`, with the same cleanup-confirmed recovery flow. That flow first discovers owner-fenced candidates, performs runtime cleanup for every expired active-generation candidate, and only then invokes the store-side repair helper before any reaping can destroy reclaimable rows. Runtime cleanup must call `runtime.Destroy(ctx, session.RestoreID)` for the current deterministic restore/container ID (`phase3-<sessionID>` today), or an explicit orphan-container cleanup that targets the persisted generation-scoped runtime ID if the implementation changes the ID scheme. For checkpointing and active/idle generations, the same rule applies to the live generation container. If runtime cleanup cannot prove that the container is absent, abort the repair for that generation and leave its allocation/resource rows non-reclaimable rather than recording a live sandbox as destroyable. After cleanup succeeds, the store helper must atomically apply the turn/generation/session repair and append the durable recovery events in the same transaction: fail `unknown_after_ack_started` turns, requeue recoverable leased turns, fail the generation, move allocation/resource rows to `reclaimable`, delete stale active contexts, clear checkpoint/restore metadata, and leave the failed generation as the active generation so the next runnable path cold-starts generation N+1. The session status must reflect remaining work after the turn repair: move to `running_idle` when no queued/leased/running turns remain, or keep a non-terminal input-blocking state such as `running_active` when queued work still needs cold fallback. No startup or periodic maintenance path may leave `sessions.status IN ('running_active','checkpointing','checkpointed')` pointing at a failed/reclaimable generation with no live retry path.
- Bridge turn completion is part of the same contract. A failed or canceled `ack_turn_completed` is a turn-level outcome, not a session terminal state. `bridge.Processor.CompleteTurn` should durably mark the turn `failed` or `canceled`, append the durable `ack_turn_completed` event, and run the guarded session-idle CAS in the same store transaction when the completed generation is still the session's active generation and no queued, leased, or running turns remain, normally moving the session to `running_idle`. If that CAS wins, the same transaction must either append a separate durable `session.running_idle` event or embed the committed idle status effect in the durable `ack_turn_completed` payload. A returned `session_marked_idle` boolean is not enough by itself; it is only a publish hint when backed by a committed event ID or by committed status fields inside the durable completion event. The CAS must preserve today's activity semantics by setting `sessions.last_activity_at` and `sessions.updated_at` to the completion time when it wins, so checkpoint eligibility is measured from turn completion rather than submit time. Duplicate completions, stale generation completions, or completions racing with newly queued work must be no-ops and must not rewrite `last_activity_at`. `handleBridgeCompletion` must not perform a second, post-commit session status update or synthesize hub-only status changes; it should only publish returned committed events and scan based on the committed `CompleteTurn` result. A crash after `CompleteTurn` commits but before `handleBridgeCompletion` runs must leave the session already input-acceptable and replayable from durable events. Keep terminal session failure reserved for protocol/store invariant errors that make the session unsafe to continue. This transition must be idempotent: a duplicate or replayed `ack_turn_completed` from generation N after generation N+1 has started must not flip the session back to `running_idle` while work is active.
- Frontend contract: the workbench must not set a session to `failed` for retryable runtime/generation events. It should show a toast, clear any streaming rows for that turn, and leave the session status returned by the backend (`created` or `running_idle`) so the composer/API retry path remains enabled. Any event that retires checkpoint/restore state must either carry a full `ApiSession` payload or a dedicated payload with every field the reducer must apply (`status`, `updated_at`, `checkpoint_path:null`, `restore_ms:null`, and any active-generation metadata). A bare `session.running_idle` event is not sufficient because the current provider only updates `status` and `updated_at` for `session.*` events and would leave stale checkpoint metadata in memory.

### 3. Add `harness.reaper.checkpoint_image_retention`

The reaper skips `reserved_checkpointed` allocations entirely (see `docs/phase7/runtime-resources.md` reaper-visibility list). A session that goes `checkpointed` and is never touched again retains its checkpoint image and `/30` slot forever.

New config:

```yaml
harness:
  reaper:
    failed_retention: 10m
    checkpoint_image_retention: 720h  # new; time.ParseDuration-compatible 30 days
```

Semantics and validation: `checkpoint_image_retention` is a `time.ParseDuration` value in the reaper config. The default is `720h`. `0s` is valid and means immediate checkpoint-image retirement on the next maintenance pass for checkpointed sessions whose retirement anchor is before `now`; it is not a disabled-retirement sentinel. Negative values are invalid and must fail config validation. Operators who want to effectively disable checkpoint-image retirement must choose an explicitly large positive duration.

Behavior: when a `checkpointed` active generation's owning session has `retirement_anchor < now - checkpoint_image_retention`, the Phase 7 maintenance job atomically retires that checkpoint metadata before any allocation rows become reclaimable. Define `retirement_anchor` as `COALESCE(sessions.last_activity_at, runtime_generations.checkpoint_created_at, sessions.updated_at, sessions.created_at)` and remove the current `last_activity_at IS NOT NULL` eligibility gate. This keeps normal turn-completion activity semantics while allowing legacy/manual checkpointed rows with NULL `last_activity_at` to retire. `RetireExpiredCheckpoints` must take the current process `OwnerUUID`, begin a transaction, call `assertOwnerTx` before mutating any rows, and fail without partial updates on an owner mismatch. The transaction must:

- CAS the session from `status = 'checkpointed'` and `active_generation_id = generation_id` to an input-acceptable non-checkpointed state, normally `running_idle`, while preserving the session row, messages, and workspace. Always clear API-visible checkpoint metadata on the session row in the same transaction, including `checkpoint_path` and any restore bookkeeping that would make the workbench or restore path believe a deleted checkpoint is still usable.
- CAS the owning `runtime_generations` row from `checkpointed` to `failed` with `error_class = 'checkpoint_retired'`, a clear `failure_reason`, `ended_at = now`, and no live lease owner.
- Move matching `network_profiles.allocation_state` and `runtime_generation_resources.resource_state` from `reserved_checkpointed` to `reclaimable`.
- Append the durable event row that tells subscribers about the committed status and metadata effect. Use either a full-session payload on the resulting status event or a dedicated non-terminal event such as `session.checkpoint_retired` with fields the reducer applies directly: `{terminal:false, generation_id, session_status:"running_idle", checkpoint_path:null, restore_ms:null, active_generation_id}` plus `updated_at`/`last_activity_at` as committed. Do not publish a bare `session.running_idle` event for this transition unless the frontend reducer has an explicit full-session refresh path for that event. The event append must be in this transaction, with a deterministic dedupe key. `restore_id` is the stable runtime handle today and should not be nulled unless the implementation first changes its semantics to checkpoint-specific metadata.

After a successful retirement commits, maintenance must publish the committed event ID for each retired session. It must not append a new status event after commit or synthesize a hub-only status change. Live SSE and replay must both clear stale `checkpointed` UI state, including `checkpoint_path` and restore metadata, from the same durable event that reflects the committed database state.

Checkpoint-retired generations must become physically destroyable on the same maintenance pass that retires them. The current destroy eligibility query in `ListDestroyableReclaimableGenerations` gates failed generations behind `failed_retention`; extend that query or add an equivalent cleanup path so `g.status = 'failed' AND g.error_class = 'checkpoint_retired'` bypasses `failed_retention` once its allocation and resource rows are `reclaimable`. Ordinary failed generations must still respect `failed_retention`. Without this class-specific bypass, `checkpoint_image_retention: 0s` would retire metadata immediately but checkpoint files and `/30` slots would still lag by `failed_retention`.

The same candidate query currently requires `n.netns_name LIKE 'harness-gen-%'`. That makes filesystem cleanup unreachable for reclaimable rows from non-sandbox mode or rows with missing/incomplete network metadata, even though those rows may still have per-generation directories. Destroyability must be selected from allocation/resource state plus the failed-retention rules, not from sandbox netns naming. Network metadata should influence only the nft/veth/netns cleanup substep inside `DestroyGenerationResources`.

The reaper then runs the generation cleanup path and frees the network slot. That cleanup path must delete every generation-scoped filesystem artifact as well as nft/veth/netns: `CheckpointPath`, `ControlDirPath`, `BundleDirPath`, `BridgeDirPath`, and `LogDirPath`. Expanding `runtime.DestroyGenerationResources` is preferred; a dedicated checkpoint-image cleanup path is acceptable only if the same maintenance flow also removes the other per-generation dirs and keeps cleanup idempotent when paths are already missing.

Filesystem cleanup is mandatory for every generation cleanup attempt and must not be gated on sandbox networking. `DestroyGenerationResources` should always attempt validated filesystem deletion first or in an independent phase. Only nft/veth/netns cleanup is conditional on `runsc_network == sandbox`; non-sandbox runtime rows still need their per-generation directories removed. Empty/missing netns or veth metadata must not short-circuit filesystem cleanup. Missing filesystem paths are successful/idempotent cleanup; invalid filesystem paths are hard errors and must prevent `MarkGenerationResourcesDestroyed`.

Recursive deletion safety is required because cleanup paths come from DB-backed `RuntimeGenerationDetails`. Before any `os.RemoveAll`, the implementation must validate the cleaned absolute target against the configured root for that resource and `details.GenerationID`. The allowed delete targets are the known per-generation locations for that generation only: `<run_dir>/control/gen-<generation_id>`, `<run_dir>/runtime/gen-<generation_id>`, `<run_dir>/bridge/gen-<generation_id>`, `<run_dir>/logs/gen-<generation_id>`, and the checkpoint path under `<run_dir>/gen-<generation_id>/checkpoint` (or the exact derived checkpoint generation root if the implementation chooses to delete that root). These targets must stay under the configured `Phase7RunDir`, include a `gen-<generation_id>` generation directory component, and match the expected derived location for their resource type.

Legacy checkpoint fallback is a separate, explicitly allow-listed cleanup case. The runtime still supports pre-P0 checkpoint images under `CheckpointsRoot/<session_id>`, so the P0 migration/backfill must preserve the real legacy checkpoint path on `runtime_generation_resources.checkpoint_path` instead of overwriting it with a generated per-generation path. `DestroyGenerationResources` may delete a legacy checkpoint target only when the stored checkpoint path cleans to exactly `<CheckpointsRoot>/<session_id>` for the owning session, the path stays under the configured `CheckpointsRoot`, and the row is being destroyed for that same active retired generation. New checkpoints must use the Phase 7 run-dir layout; the legacy allow-list exists only to drain old rows. Reject empty paths, `..` escapes, symlink-resolved escapes where the target exists, mismatched generation/session IDs, paths outside the appropriate configured root, and arbitrary checkpoint paths not matching either the Phase 7 derived location or the legacy `CheckpointsRoot/<session_id>` pattern.

The next user message must cold-start, not restore: `ensureActiveGeneration` should see the active generation as `failed` or absent, allocate generation N+1, and update `sessions.active_generation_id` through the existing allocation CAS. A direct `reserved_checkpointed -> reclaimable` flip without retiring the active checkpointed generation is invalid because `ensureActiveGeneration` would attempt restore and fail the allocation/resource CAS.

Race handling is part of this requirement. `ensureActiveGeneration` can read a checkpointed session/generation and then lose the race to `RetireExpiredCheckpoints` before `ClaimCheckpointedGenerationForRestore` executes. That stale restore CAS must be treated as a retryable refetch, not a 500: refetch the session/generation state once and cold-start if the active generation has been retired to `failed` and the session is now non-checkpointed. Only non-stale CAS failures that indicate corruption or broken invariants should bubble out as server errors.

## P0 Release Constraint: `max_sessions`

`session_retention: 0s` intentionally stops automatic destruction of active session rows. P0 does not change `CountActiveSessions`: `harness.max_sessions` remains a soft ceiling on non-terminal sessions (`created`, `running_active`, `running_idle`, `checkpointing`, `checkpointed`), not a count of live gVisor resources. A lab can hit the ceiling even after checkpoint/runtime resources have been retired, because historical retryable sessions still count.

Because `max_sessions` is no longer a live-resource ceiling, it must not be validated against `/30` network-slot capacity. The existing config check at `orchestrator/internal/config/config.go:424` must be removed or replaced with validation that only checks `harness.max_sessions > 0` and validates the CIDR pool shape independently. Pool exhaustion belongs to allocation-time checks: generation allocation should fail/retry cleanly when no `/30` slot is available, while config loading must allow operators to set `max_sessions` higher than the live runtime pool to retain long-lived non-terminal history. The quota API should report the two ceilings separately instead of clamping an `effective_ceiling` to the smaller value.

Required P0 handling:

- Document the constraint in operator docs and release notes next to `session_retention` and `max_sessions`: operators who want long-lived history must either set `harness.max_sessions` high enough for retained non-terminal history or close sessions explicitly.
- Update config validation and tests so `harness.max_sessions` may exceed `/30` pool capacity, while allocation-time resource exhaustion still returns a retryable runtime/allocation error rather than accepting more live sandboxes than the pool can provide.
- Keep the existing manual close path (`DELETE /api/sessions/{id}` / `DestroySession`) available and document that it preserves the session row, messages, artifacts, workspace, and agent home, while marking the session terminal and reclaiming runtime resources. For P0, close is an immediate cleanup operation, not a best-effort mark plus future maintenance. The handler must destroy the live runtime container for the active generation/restore ID, then run a store transaction that cancels pending turns, marks the session `destroyed`, fails all non-terminal generations with `session_destroyed`, makes allocation/resource rows reclaimable, clears active contexts and checkpoint/restore metadata, appends the durable terminal close event, and returns the reclaimed generation IDs. It must then run the same generation resource cleanup path used by checkpoint retirement (`DestroyGenerationResources` plus `MarkGenerationResourcesDestroyed`) for those generations before returning success, so `/30` slots, nft/veth/netns, checkpoint images, and per-generation filesystem directories are freed immediately when cleanup succeeds. If the UI/operator needs to distinguish terminal close from completed physical cleanup, append and publish a separate durable cleanup-complete event only after `MarkGenerationResourcesDestroyed` commits. If live runtime deletion or generation resource cleanup fails, the API must return an error or an explicit partial-cleanup response and must not publish a durable event that implies resources were fully reclaimed. Closed sessions no longer count toward `max_sessions`, but they are not input-acceptable again until the future resume UX exists.
- Expose a user/operator close action in the workbench before shipping the `0s` default, so hitting `max_sessions` is recoverable without direct API calls. The action may use the existing `destroyed` status for P0; adding a separate `archived` status or resume-from-closed flow remains out of scope.
- Do not implement automatic archival in P0. Automatic close policies can destroy retryability unexpectedly and need separate product semantics.

## Out of Scope (Follow-up)

- A "resume failed/destroyed session" UX path. The data is recoverable; the API gates on `sessionstate.CanAcceptInput`. A future change can add `POST /api/sessions/<id>/resume` that resets the session status to `running_idle` and clears `active_generation_id`. Tracked as a follow-up, not a P0 blocker.
- Workspace artifact GC policy. Workspaces grow forever today. Disk pressure is bounded by deployment scale for now.
- Changing `CountActiveSessions` semantics beyond the explicit P0 release constraint above. A future change can add `archived` as a first-class non-active status, separate active-session quota from historical-session quota, or add resume-from-closed behavior.
